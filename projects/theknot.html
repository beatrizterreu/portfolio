The role

Design System Lead is a hybrid role that sits at the intersection of design, product ownership, and strategy. My work goes beyond defining visual language: it includes architecture decisions, prioritisation of system initiatives, cross-functional alignment between design, engineering, and product, and leading adoption processes across teams.

As Product Owner of ODS, I define the system's direction, evaluate impact across different products and contexts, and ensure every decision has measurable value for the teams that depend on it. I balance long-term vision with day-to-day execution — working hands-on in Figma while simultaneously managing governance, roadmap, and stakeholder communication.

One system, two identities. One token architecture, zero compromise on brand fidelity.

The challenge

The Knot Worldwide operates a portfolio of wedding platforms across 16+ markets. Two primary brands drive the product experience: The Knot — vibrant, modern, US-led — and Hitched, a more understated UK brand with its own distinct identity. Both had evolved independently, leading to duplicated effort, inconsistent experiences, and a design system that couldn't scale across brands or platforms without significant rework.

The challenge was structural: evolve the design system into a true multi-brand model — capable of supporting both brands, and future ones, from a single architecture — without creating a lowest-common-denominator experience or losing what makes each brand distinctive.

Token architecture

The foundation of ODS is a three-layer token system built on the W3C DTCG format. Each layer has a specific responsibility, and the chain flows in one direction only.

Layer 1
Primitive tokens
Raw values. Brand-specific, non-semantic. Source of truth for all color, size, and spacing. Each brand defines its own set.
pColor.primary.300 pColor.neutral.500 pSize.48 pSpace.16 pBorderRadius.full pFontSize.16
→
Layer 2
Semantic tokens
Intent-based aliases. Consume primitives and expose contextual meaning: text, background, border, icon, focus.
sColor.text.brand sColor.background.light sColor.border.error sColor.icon.disabled sColor.focus.default
→
Layer 3
Component tokens
Component-scoped decisions. Each component owns its states: default, hover, pressed, disabled, error, success.
cButton.primary.fill.default cInput.border.error cBadge.fill.callOut cCard.background

This architecture means changing a brand only requires updating its primitives. The semantic and component layers adapt automatically — without touching a single component. Switching from The Knot magenta to Hitched purple is a token swap, not a redesign.

Two brands, one system

The Knot and Hitched are built on the same skeleton. What differs is how they express it. The semantic layer is where the system decouples brand identity from structural behaviour: sColor.text.brand maps to pColor.primary.400 for The Knot (#d700a0) but to pColor.primary.300 for Hitched (#775a97). Different values, same intent, same component.

The Knot

Hitched

Platform strategy: Web and iOS

ODS serves both Web and iOS from the same token base, while accommodating the structural differences each platform demands. 18 components are shared across both platforms. 10 are iOS-exclusive (TabBar, PillBar, NavigationBar, Toggle, StarsRating, DrawerPresentation...). 10 are web-exclusive (SubNav, SideNav, DatePicker, Carousel, Pagination, Breadcrumb...).

Platform deltas are documented and intentional. cButton.size.lg is 48px on web, 50px on iOS. cInput.height is 48px on web, 44px on iOS. cOverlay.opacity.dark is 70% on web (visual depth layer) vs 20% on iOS (native navigation convention). These are not inconsistencies — they are platform-appropriate decisions baked into the token structure, traceable from Figma to production.

Design to code pipeline

The pipeline enables a single change in Figma to propagate automatically across all four surfaces: The Knot Web, The Knot iOS, Hitched Web, and Hitched iOS.

01
Figma Variables
Tokens defined as Figma variables using the W3C DTCG format. Scopes set to ALL_SCOPES for maximum flexibility.
$type · $value · ALL_SCOPES
02
JSON Export
Tokens Studio exports brand files preserving Figma IDs and token references intact. The _themes.json maps each set to its platform group.
_themes.json · web / ios groups
03
GitHub Repo
One JSON file per brand and platform combination. References resolved at build time by Style Dictionary or the consuming app.
01_web-theKnot.json · 03_web-hitched.json
04
Platform Consume
Web reads via CSS custom properties. iOS Swift generates SwiftUI-compatible token values. One Figma change. Four surfaces updated.
CSS vars · SwiftUI tokens

Beyond the system

Owning a design system at this scale means owning its adoption, not just its components. I define how teams work with the system, what governance looks like, and how we manage change without disrupting teams that depend on stability. This includes frameworks for contribution, decision logs, and progressive rollout strategies that reduce friction and build trust.

As the bridge between design, engineering, and product, I translate architectural complexity into clear decisions and actionable guidance. When a team asks "which token do I use here?" the answer should be obvious — and when it isn't, that's a signal to improve the system, not just the documentation.

What I learned

The hardest problem in multi-brand system design is not the tokens — it is the governance. Deciding which differences are brand features and which are inconsistencies requires organizational clarity that goes well beyond Figma. The technical architecture is the easy part; building shared language between design and engineering teams takes far more time and care.

The three-layer token architecture is not just a naming convention. It is a decision-making framework. When the layers are clear, every design question has a natural home: is this change to a primitive value, a semantic alias, or a component-scoped decision? That clarity accelerates work, reduces unintended side effects, and makes the system genuinely scalable — not just today, but as the product grows.
